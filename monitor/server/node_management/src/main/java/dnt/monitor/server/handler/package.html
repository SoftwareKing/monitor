系统模型分为三个层面：

  1. 资源: Resource/Link/Component 等对象
  2. 管理: ManagedNode
  3. 展示: TopoMap/TopoNode/TopoLink
系统的可能从三个层面的任一层面发起模型更新，如：
  • 用户会通过界面，从管理层开始发起模型更新任务（创建资源/引擎/分组等)
  • 用户也可能会通过界面的topo模块，为展示美观单纯修改topo层信息（此时往往不会涉及到管理/资源的改动）
  • 引擎会通过资源层面向服务器发起各种模型更新任务，如引擎注册，修改，注销；资源发现，更新等
为了解决系统模型更新，确立一致性原则：

操作的顺序原则为：

创建时:
  1. 资源
  2. 管理
  3. 展示
删除时:
  1. 展示
  2. 管理
  3. 资源
修改时:
  1. 先被修改的目标层面
  2. 而后是其他层面，其他层面的更新原则是先底层，后上层
例如：当用户通过界面，修改管理层信息时，有资源层和展示层需要更新，此时应该先更新资源层，再更新展示层； 当资源层发生更新后，有管理层和展示层需要更新，那么应该先更新管理层，再更新展示层（且应该通过管理层自身的更新时间）
